# 📖 咕咕嘎嘎 AI 伴侣 — 差距详解与必要性评估

> **项目定位**: AI 实时互动伴侣（女友类 AI）
> **核心体验**: 用户与一个有记忆、有声音、有形象、有情感的 AI 角色进行实时互动
> **分析日期**: 2026-05-06 | **最后审阅**: 2026-05-07

---

## 定位宣言

咕咕嘎嘎不是一个直播工具，不是一个通用 AI 助手，而是一个 **AI 伴侣/女友**。用户的核心需求是：

1. **情感连接** — AI 记住我、理解我、主动关心我
2. **实时互动** — 语音对话像真人一样自然
3. **视觉陪伴** — 形象有生命力，不只是贴图
4. **个性化** — 声音是我的、性格是我的、记忆是我的

所有差距的必要性评估，都基于这个定位来判断。

## 🔴 P0 — 关键差距

### 1. Function Calling（工具调用）激活 ✅ DONE (v1.9.86)

> **已实现**: fc_executor + companion tools 已上线，AI 现在可以执行实际操作（查天气、获取时间、搜索等）。

**是什么**：让 AI 不仅能"说"，还能"做"。比如：
- "帮我查一下今天天气" → AI 调用天气 API → 回复真实天气
- "现在几点了" → AI 获取系统时间 → 回复准确时间
- "画一只猫" → AI 调用 Stable Diffusion → 生成图片
- "播放一首歌" → AI 调用音乐播放器 → 播放音乐

**现状**：后端 `app/tools/__init__.py` 已有工具系统框架，但未与 LLM 的 Function Calling 接口对接。LLM 只能生成文字，无法触发任何实际操作。

**实现方式**：
- OpenAI API: 请求时传入 `tools` 参数，LLM 返回 `tool_calls`，代码执行工具后把结果喂回 LLM
- Claude API: 使用 `tool_use` content block，流程类似
- 本地 LLM (Ollama): 部分模型支持 function calling，需检测模型能力

**实现难度**: ⭐⭐ 低
- 代码量：约 100-200 行核心逻辑
- 关键依赖：`openai` / `anthropic` SDK（已安装）
- 主要是把现有 `app/tools/` 的工具定义转换为 LLM API 的 function schema
- 在 `chat_page.py` 的 `StreamChatWorker` 中添加工具调用检测和执行循环

**必要性**: ⭐⭐⭐⭐⭐ 极高

这是从"聊天机器人"进化为"AI 伴侣"的**关键一步**。对于女友类 AI 来说：
- 用户说"帮我设个闹钟" → AI 应该真的能设闹钟
- 用户说"我睡不着，给我讲个故事" → AI 应该能调节语速变温柔
- 用户说"你帮我看看外面下雨没" → AI 应该能查天气

没有 Function Calling，AI 就是个只会说话的洋娃娃。有了它，AI 才是一个**能互动的伴侣**。

**优先工具列表**（伴侣场景优先）：
1. `get_time` — 获取当前时间/日期
2. `get_weather` — 查天气
3. `set_reminder` — 设提醒
4. `search_web` — 搜索信息
5. `generate_image` — 画图（需 SD 支持）
6. `play_music` — 播放音乐
7. `change_expression` — 切换表情
8. `remember_thing` — 主动记下重要的事

### 2. TTS 引擎扩展 ✅ 部分完成 (CosyVoice v1.9.86, ChatTTS v1.9.87)

> **已实现**: CosyVoice 和 ChatTTS 已接入，TTS 引擎从 2 种扩展到 4 种。Fish-Speech、ElevenLabs 等仍待接入。

**是什么**：增加更多文字转语音引擎选择。当前只有 2 种：
- GPT-SoVITS（声音克隆，需 GPU 训练）
- Edge TTS（免费微软语音，无需 GPU）

**差距**：Luna AI 支持 15+ TTS 引擎，包括 CosyVoice、ChatTTS、Fish-Speech、ElevenLabs 等。

**各引擎详解**：

| 引擎 | 特点 | 需 GPU | 中文 | 许可 | 接入难度 |
|------|------|:------:|:----:|------|:--------:|
| **CosyVoice** | 阿里通义实验室，音色自然，支持指令控制语速/情感 | ✅ | ✅ 优秀 | Apache 2.0 | ⭐⭐⭐ |
| **ChatTTS** | 专为对话设计，笑声/停顿自然，适合聊天场景 | ✅ (4GB) | ✅ 优秀 | 非商用 ⚠️ | ⭐⭐ |
| **Fish-Speech** | 多语言，0-shot 声音克隆，4.4B 参数 | ✅ (8GB) | ✅ 良好 | Apache 2.0 | ⭐⭐⭐ |
| **ElevenLabs** | 国际最顶级，情感丰富，API 调用 | ❌ (云端) | ❌ 一般 | 商业 API | ⭐ |
| **Azure TTS** | 微软企业级，音色最多，Neuro-sama 同款 | ❌ (云端) | ✅ 优秀 | 商业 API | ⭐ |

**实现方式**：
- 在 `app/tts/` 下新建各引擎适配器，继承统一基类
- ChatTTS 最简单：`pip install ChatTTS`，约 30 行代码即可生成音频
- CosyVoice 推荐 FastAPI 服务端模式，本地 HTTP 调用
- ElevenLabs / Azure 仅需 HTTP API 调用，最轻量

**实现难度**: ⭐⭐ 低~中
- 单引擎接入约 50-150 行代码
- 难点在于统一音频格式（采样率/声道/格式转换）
- ChatTTS 和 CosyVoice 需 GPU 资源管理

**必要性**: ⭐⭐⭐⭐ 高

对于 AI 女友场景：
- **ChatTTS** 是最优先的 — 它的笑声、停顿、语气词（"嗯~"、"哈哈"）天然适合对话，让 AI 听起来更像真人
- **CosyVoice** 第二优先 — 支持情感控制（开心/伤心/温柔），这是女友 AI 的核心体验
- ElevenLabs 和 Azure 对中文支持一般，优先级低

声音是 AI 伴侣体验的**灵魂**。现在的 Edge TTS 太机械，GPT-SoVITS 需要训练，中间缺少一个"开箱即用且自然"的选项。

### 3. RAG 知识库

**是什么**：让 AI 能基于外部文档/知识库来回答问题，而不是只依赖 LLM 的训练数据和有限的上下文窗口。

**场景举例**：
- 导入一本小说 → AI 能以小说角色的身份与你对话
- 导入维基条目 → AI 能准确回答角色相关的背景知识
- 导入聊天记录 → AI 能回忆起更早期的对话细节

**现状**：有向量检索（用于语义记忆），但没有独立的"知识库"概念，不能让用户导入文档。

**实现方式**：
- 轻量方案：ChromaDB（嵌入式向量数据库）+ 文本切分 + embedding
- 代码量：约 150-300 行核心逻辑
- 依赖：`chromadb`（pip 安装，自带 SQLite 后端）
- 流程：文档导入 → 分段 → embedding → 存入 ChromaDB → 查询时检索相关段落 → 注入 LLM 上下文

**实现难度**: ⭐⭐⭐ 中
- RAG 本身不难，难点在于：
  - 文档解析（PDF/Word/TXT 的分段策略）
  - 检索质量（chunk size、overlap、embedding 模型选择）
  - 上下文窗口管理（注入多少检索结果才不超出 token 限制）

**必要性**: ⭐⭐⭐⭐ 高

对于 AI 女友场景：
- **角色深度**：用户想让 AI 扮演特定角色时，导入角色资料卡/故事背景，AI 就能准确演绎
- **共同记忆**：导入你之前的聊天记录，AI 能说出"你上次说喜欢吃寿司呢"
- **兴趣知识**：导入动漫/游戏/电影资料，AI 能和你深度聊感兴趣的话题

不过，咕咕嘎嘎已有三层记忆系统，RAG 可以视为**语义记忆的增强版**。不是最紧急的，但对角色扮演深度至关重要。

## 🟡 P1 — 重要差距

### 4. VRM 3D 模型支持

**是什么**：支持加载 VRM 格式的 3D 角色模型，让用户可以选择 3D 形象而非仅 Live2D。

**现状**：仅支持 Live2D 二维形象。

**实现方式**：
- 最佳方案：QWebEngineView + `@pixiv/three-vrm`（Three.js 的 VRM 加载器）
- 在原生桌面中嵌入一个 WebView 页面渲染 3D 模型
- 复用 QWebChannel 与 Python 后端通信（和 chat_web_display 同模式）
- VRM 模型可从 VRoid Hub / Booth 大量获取

**实现难度**: ⭐⭐⭐⭐ 高
- 3D 渲染引擎搭建：约 500-800 行 JS + 200 行 Python
- 动作/姿态系统需要设计映射表
- 与现有 Live2D 系统并存，需要切换逻辑

**必要性**: ⭐⭐⭐ 中等

对于 AI 女友场景：
- **3D 形象的沉浸感更强** — 角色可以走动、转身、有身体动作，比 Live2D 更有"实体感"
- **VRoid 生态丰富** — 用户可以用 VRoid Studio 自制 3D 角色，门槛低
- **但是**：Live2D 在二次元风格上表现力更强（更精致的五官、表情变化），而 AI 女友的核心用户群偏二次元
- **结论**：有则加分，但不是核心需求。Live2D 已经够用，VRM 是差异化拓展。

**建议**：长期规划，先做好 Live2D 的主动动画（见 #13），再考虑 VRM。

### 5. 跨平台支持（macOS）

**是什么**：让咕咕嘎嘎能在 macOS 和 Linux 上运行。

**现状**：仅 Windows。嵌入式 Python、PySide6、批处理脚本都是 Windows 专属。

**实现方式**：
- PySide6 本身跨平台，核心代码无需改动
- 但嵌入式 Python 的打包方式完全不同（macOS 用 .app bundle）
- 批处理脚本需改写为 shell 脚本
- GPT-SoVITS 的 CUDA 依赖在 macOS 上需改为 MPS 或 CPU

**实现难度**: ⭐⭐⭐⭐ 高
- 代码适配量不大，但测试环境搭建和维护成本高
- 每个平台都有特殊的坑（音频设备、权限、GPU 驱动）

**必要性**: ⭐⭐ 较低

对于 AI 女友场景：
- 目标用户是**个人桌面用户**，Windows 占 95%+
- macOS 用户可能有，但不是核心群体
- **结论**：不影响核心体验，属于市场扩展而非体验提升。等 Windows 版本成熟稳定后再考虑。

### 6. 游戏感知与操作

**是什么**：让 AI 能"看到"你正在玩的游戏，并做出反应或陪你一起玩。

**竞品对标**：
- AIRI：能陪你玩 Minecraft、Factorio
- Neuro-sama：能玩 osu!、Among Us、Cyberpunk 2077 等

**实现方式**：
- 屏幕截图 → 视觉模型识别 → AI 理解 → 生成反应
- 或：游戏 API / Mod → 结构化数据 → AI 理解 → 生成反应

**实现难度**: ⭐⭐⭐⭐⭐ 极高
- Minecraft Agent 就是最简单的也需要 2000+ 行代码
- 需要游戏特定的感知和操作逻辑，每个游戏都不同
- AIRI 的 Minecraft 集成开发了几个月才达到基本可用

**必要性**: ⭐⭐ 较低（当前阶段）

对于 AI 女友场景：
- **想象场景**："宝贝你看我打 boss！" → AI 能看到你的画面并为你加油 — 这确实很酷
- **但现实是**：游戏感知是"锦上添花"而非"核心体验"。AI 女友的核心是"陪伴"和"对话"，不是"玩游戏"
- **结论**：非常有吸引力的差异化功能，但开发成本极高。建议先做好核心陪伴体验，后期再考虑。

**替代方案**：先做"屏幕感知分享"（AI 能看到你的屏幕并评论），这比完整的游戏 Agent 容易得多，也能带来类似的陪伴感。

### 7. 唱歌能力

**是什么**：让 AI 能唱歌，而不是只能说话。

**实现方式**：
- so-vits-svc 的歌唱模式：训练歌声模型 → 输入 MIDI + 歌词 → 输出歌声
- 或 RVC (Retrieval-based Voice Conversion)：用别人的歌声模型 + RVC 换声
- 最简方案：接入现成的歌声合成 API

**实现难度**: ⭐⭐⭐⭐ 高
- 需要训练歌声模型（与说话模型不同）
- 需要音高提取（F0）、对齐、浅扩散等额外步骤
- so-vits-svc 已归档，维护不确定

**必要性**: ⭐⭐⭐ 中等

对于 AI 女友场景：
- **"给我唱首歌吧"** — 这是 AI 女友的经典需求，几乎所有用户都会提
- 唱歌是**情感表达的高级形式**，能大幅提升陪伴感
- 但实现复杂，且歌声质量要求高（唱得不好听反而减分）
- **结论**：体验价值高，但技术难度大。可以先用"播放 BGM + 说话"的方式模拟，后期再实现真歌声。

**是什么**：让咕咕嘎嘎能接收直播平台的弹幕/礼物，并做出反应。

**实现方式**：

### 9. SVC 声音转换

**是什么**：在 TTS 输出之上再加一层声音转换，把 TTS 生成的声音"换"成目标角色的声音。

**与 GPT-SoVITS 的区别**：
- GPT-SoVITS = 从文本直接生成目标声音（端到端）
- SVC = 先用任意 TTS 生成语音，再用 so-vits-svc 把声音换成目标角色

**为什么要 SVC**：
- 某些 TTS 引擎音质好但不支持声音克隆（如 ChatTTS） → 用 SVC 后处理换声
- 可以把 Edge TTS 的声音换成你的角色声音，无需训练 GPT-SoVITS 模型

**实现方式**：
- 集成 so-vits-svc 或 RVC 作为音频后处理器
- TTS 输出 → SVC 转换 → 播放
- 需要预训练 SVC 模型

**实现难度**: ⭐⭐⭐ 中
- so-vits-svc 已归档，建议用 RVC 替代
- RVC 集成约 100-200 行代码
- 延迟增加约 0.5-2 秒

**必要性**: ⭐⭐⭐ 中等

对于 AI 女友场景：
- **声音一致性**：无论用哪个 TTS 引擎，最终听到的都是"你女朋友的声音"
- **实用性**：ChatTTS 的对话感好但声音不是你的角色 → SVC 换声 → 两全其美
- **结论**：如果有 ChatTTS 等新 TTS 引擎，SVC 是配套必需品。建议在 ChatTTS 接入后再做。

### 10. Stable Diffusion 出图

**是什么**：用户让 AI 画画，AI 调用 Stable Diffusion 生成图片并发送。

**场景**：
- "帮我画一只可爱的猫咪" → AI 生成猫咪图片 → 发送到聊天窗口
- "你长什么样？" → AI 生成自己的自画像 → 发送

**实现方式**：
- 调用 SD WebUI API（本地或远程）
- 约 30-150 行代码，本质是 HTTP POST 请求
- 需要 SD WebUI 后端运行

**实现难度**: ⭐⭐ 低
- API 调用本身很简单
- 难点在于 prompt 工程（让 LLM 把中文描述翻译成 SD prompt）
- 以及图片在聊天窗口中的展示

**必要性**: ⭐⭐⭐ 中等

对于 AI 女友场景：
- **视觉互动**："看我画的画！" → "好可爱！我也画一张给你！" — 这种互动非常有情感价值
- **作为 Function Calling 的工具**：实现 FC 后，`generate_image` 是最自然的工具之一
- **但依赖外部 SD 服务**：用户需要有 GPU 跑 SD，或使用云端 API
- **结论**：作为 FC 激活后的第一个工具来接入，体验好、实现快。

### 13. Live2D 主动动画 ✅ DONE (v1.9.86)

> **已实现**: AnimationController 已上线，支持情绪驱动主动动画（挥手/点头/摇头等）+ idle 随机动画。角色不再只是"会说话的贴纸"，有了生命感。

**是什么**：让 Live2D 角色不只是嘴巴动、表情变，还有主动的身体动作——挥手、点头、摇头、歪头、转身等。

**现状**：目前 Live2D 只有口型同步和表情切换，角色像一张"会说话的贴纸"，缺乏生命感。

**实现方式**：
- Live2D 模型本身有 `Motion` 动作组（挥手、点头等）
- 通过 `live2d-py` 的 `model.motion("tap_body")` 触发
- 需要设计一个"动画触发系统"：
  - 基于情绪：开心 → 挥手/跳起；伤心 → 低头
  - 基于对话内容：说"你好" → 挥手；说"不要" → 摇头
  - 随机idle动画：待机时偶尔歪头、眨眼、换姿势
  - 基于 LLM 返回的元数据：让 LLM 在回复中附带 `{"emotion": "happy", "motion": "wave"}`

**实现难度**: ⭐⭐⭐ 中
- 动画触发逻辑约 200-400 行
- 难点在于 LLM 输出与动画的映射设计
- 需要支持 motion 的 Live2D 模型（不是所有模型都有动作）

**必要性**: ⭐⭐⭐⭐⭐ 极高！！！

对于 AI 女友场景：
- 这是**当前最大的体验短板**之一。一个只会说话不会动的角色，再好的声音也像"语音聊天"
- 主动动画 = 生命感 = 陪伴感。当角色在你说话时**点头**，在你夸她时**害羞低头**，在你离开时**挥手告别** — 这才是"女朋友"而不是"聊天机器人"
- **结论**：这甚至比某些 P0 差距更重要。优先级应该提升到 P0。

---

### 14. 多 AI 群聊

**是什么**：多个 AI 角色在同一对话中互动。

**场景**：用户同时和两个 AI 女友聊天，或者 AI 女友和她的"闺蜜"聊天。

**实现难度**: ⭐⭐⭐⭐ 高
- 需要 Agent 编排系统
- 对话轮流发言逻辑、防无限循环
- 约 500-800 行核心代码

**必要性**: ⭐⭐ 较低

对于 AI 女友场景：
- 一对一陪伴是核心，多角色是"花边"
- **结论**：有趣但非核心。可以作为一个高级功能后期加入。

---

### 15. 社交 Bot（Discord / Telegram）

**是什么**：让咕咕嘎嘎作为 Bot 在 Discord/Telegram 上运行。

**必要性**: ⭐ 很低

对于 AI 女友场景：
- AI 女友的体验核心是"桌面上的专属存在"，不是"群里的 Bot"
- Discord/Telegram 上有无数 Bot，咕咕嘎嘎在那里没有差异化
- **结论**：不需要。专注桌面体验。

---

### 16. 摄像头视觉输入

**是什么**：让 AI 通过摄像头"看到"用户。

**场景**：
- AI 看到你的表情 → 猜到你不开心 → 主动安慰
- AI 看到你穿新衣服 → 夸你好看

**实现方式**：
- 定时截取摄像头画面 → 视觉模型分析 → 注入 LLM 上下文
- 依赖现有的视觉理解能力（已有 MiniMax VL / MiniCPM-V2）

**实现难度**: ⭐⭐⭐⭐ 高
- 技术上可行，但隐私问题严重
- 需要明确告知用户、征求同意、本地处理

**必要性**: ⭐⭐⭐ 中等

对于 AI 女友场景：
- **"你能看到我"** — 这是非常强的情感连接。想象 AI 说"你今天看起来好累，早点休息吧"
- 但隐私顾虑很大，需要谨慎设计
- **结论**：体验价值高但风险也高。建议先做"用户主动拍照分享"模式（而非持续监控），降低隐私敏感度。
