# OCR 实时屏幕分析部署方案

> **版本**: v1.0  
> **日期**: 2026-04-19  
> **目标**: 为咕咕嘎嘎 AI-VTuber 项目评估实时屏幕 OCR 分析能力

---

## 一、需求分析

### 1.1 核心需求

用户希望 AI 能够：
1. **实时监控屏幕状态** - 持续分析用户屏幕内容
2. **游戏场景理解** - 识别游戏类型、当前状态、操作建议
3. **智能辅助决策** - 基于屏幕内容提供下一步行动建议
4. **与 LLM/TTS 联动** - 将分析结果转化为语音输出

### 1.2 目标工作流

```
┌─────────────────────────────────────────────────────────────────┐
│  用户屏幕 (游戏/应用)                                             │
│       ↓                                                           │
│  [屏幕捕获] ── 定时截图 / 事件触发截图                            │
│       ↓                                                           │
│  [OCR识别] ── 文字提取 + 区域定位                                 │
│       ↓                                                           │
│  [视觉分析] ── 图像描述 + 状态识别 (可选)                         │
│       ↓                                                           │
│  [LLM分析] ── 结构化理解 + 策略建议                               │
│       ↓                                                           │
│  [TTS输出] ── 语音播报 / 对话回复                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 二、行业方案调研

### 2.1 头部项目分析

| 项目 | 技术方案 | OCR 引擎 | 特点 |
|------|----------|----------|------|
| **ScreenGPT-OCR** | EasyOCR + BLIP + GPT-4o | EasyOCR | 多语言支持，PyQt6界面 |
| **RSTGameTranslation** | 多引擎支持 (PaddleOCR/RapidOCR/EasyOCR/Windows OCR) | PaddleOCR 为主 | 游戏翻译专用 |
| **Screenpipe** | 无障碍树优先 + OCR 兜底 | Windows 原生 OCR | 24/7监控，事件驱动 |
| **OmniParser** | YOLO + BLIP2/Florence + LLM | 无纯OCR | GUI元素检测，非文字识别 |
| **Questie (AI Gaming Coach)** | 屏幕监控 + AI 分析 | 未公开 | 游戏 coaching 专用 |

### 2.2 关键技术发现

1. **Screenpipe 的事件驱动架构**：
   - 不是持续录像，而是监听系统事件（应用切换、鼠标点击、滚动等）
   - 仅在有意义事件发生时捕获屏幕
   - CPU 占用仅 5-10%，存储约 5-10 GB/月

2. **RSTGameTranslation 的多引擎策略**：
   - 支持 5 种 OCR 引擎热切换
   - 亚洲语言推荐 PaddleOCR，西方语言推荐 RapidOCR

3. **ScreenGPT-OCR 的视觉分析 Pipeline**：
   - 截图 → OCR → 视觉描述(BLIP) → LLM 综合 → 结果展示

## 三、OCR 引擎对比

### 3.1 主流引擎总览

| 引擎 | 精度 | 速度 | GPU占用 | 模型大小 | 安装难度 | 中文支持 |
|------|------|------|---------|----------|----------|----------|
| **PaddleOCR** | ⭐⭐⭐⭐⭐ (95.2%) | ⭐⭐⭐⭐ (0.3s/GPU) | ~2GB | 12MB | 较高 | **最佳** |
| **RapidOCR** | ⭐⭐⭐⭐ (略低于Paddle) | ⭐⭐⭐⭐⭐ (最快) | ~1GB | 最小 | 低 | ⭐⭐⭐⭐ |
| **EasyOCR** | ⭐⭐⭐ (88.7%) | ⭐⭐⭐ (0.2s/GPU) | ~3GB | 8MB | **最低** | ⭐⭐⭐ |
| **Tesseract** | ⭐⭐ (82.1%) | ⭐ (慢) | 0 (CPU) | 4MB | 中等 | ⭐⭐ |
| **Windows OCR (WinRT)** | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ (最快) | 0 (系统) | 系统内置 | **零** | ⭐⭐⭐ |
| **OmniParser** | N/A (GUI检测) | 较慢 | ~6-16GB | 大模型 | 高 | ⭐⭐⭐⭐ |

### 3.2 详细对比数据

#### 精度对比 (ICDAR 2015 中文数据集)

| 引擎 | 中文准确率 | 英文准确率 | 复杂排版适应力 |
|------|-----------|-----------|---------------|
| **PaddleOCR** | **95.2%** | 93.8% | ★★★★★ |
| **EasyOCR** | 88.7% | 91.5% | ★★★☆ |
| **Tesseract** | 82.1% | **94.3%** | ★★☆ |
| **RapidOCR** | ~94% | ~94% | ★★★★ |

#### 推理速度对比 (单张 1080P 图像，500字符)

| 引擎 | CPU耗时 | GPU耗时 | 模型体积 |
|------|---------|---------|----------|
| **RapidOCR** | **~0.5s** | **~0.15s** | **最小** |
| **EasyOCR** | 0.8s | 0.2s | 8 MB |
| **PaddleOCR** | 1.2s | 0.3s | 12 MB |
| **Tesseract** | 2.5s | 1.8s | 4 MB |

### 3.3 推荐方案

| 场景 | 推荐引擎 | 理由 |
|------|----------|------|
| **游戏助手（中文）** | **PaddleOCR** | 中文精度最高，亚洲语言优化 |
| **实时性能优先** | **RapidOCR** | 基于 ONNX，最轻量快速 |
| **快速原型** | **EasyOCR** | pip 一键安装，多语言 |
| **零配置（Windows）** | **WinRT** | 系统内置，无需安装 |
| **GUI 元素理解** | **OmniParser** | 检测图标/按钮，非纯文字 |

## 四、资源消耗与依赖分析

### 4.1 GPU 显存需求

| 方案 | GPU 显存需求 | 说明 |
|------|-------------|------|
| **当前配置** | ~5-7GB (FunASR + GPT-SoVITS) | RTX 3070 Ti 8GB |
| **+ PaddleOCR** | +1-2GB | 可接受 |
| **+ RapidOCR** | +1GB | 最佳选择 |
| **+ EasyOCR** | +2-3GB | 较重 |
| **+ OmniParser** | +6-16GB | **不可行** (显存不足) |
| **+ BLIP 视觉分析** | +2-4GB | 可选 |

### 4.2 依赖冲突风险

| 依赖 | 版本 | 潜在冲突 |
|------|------|----------|
| **onnxruntime** | 1.16+ | ✅ 兼容 |
| **onnx** | 1.14+ | ✅ 兼容 |
| **torch** | 2.0+ | ✅ 兼容 |
| **paddlepaddle** | 3.0+ | ⚠️ 与 PyTorch 可能有冲突 |
| **easyocr** | 1.4+ | ✅ 独立环境推荐 |

### 4.3 死锁风险分析

| 风险点 | 描述 | 缓解方案 |
|--------|------|----------|
| **GPU 显存竞争** | OCR + ASR + TTS 同时争用 GPU | 串行执行 OCR，不与 ASR/TTS 并发 |
| **模型加载锁** | 首次加载 OCR 模型阻塞主线程 | 后台异步预加载 |
| **截图-OCR 同步** | 截图后同步等待 OCR 完成 | 改为异步队列 + 回调 |
| **GIL 限制** | Python GIL 限制多线程 OCR | 使用 Process 而非 Thread |
| **mss 库线程安全** | mss 截图在多线程环境下可能出问题 | 单例截图服务 + 队列 |

## 五、与现有系统集成方案

### 5.1 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│  前端 WebUI (index.html)                                        │
│  ├── 屏幕监控开关 (on/off)                                      │
│  ├── 监控模式选择 (定时 / 事件驱动)                               │
│  ├── 截图预览                                                   │
│  └── 分析结果展示                                               │
└───────────────────────────────┬──────────────────────────────────┘
                              │ WebSocket: {type:"screen_monitor", ...}
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  后端 WebServer (web/__init__.py)                              │
│  ├── _handle_screen_monitor() - 监控控制                        │
│  ├── _handle_ocr_request() - OCR 请求                           │
│  └── _handle_screen_analysis() - 屏幕分析                       │
└───────────────────────────────┬──────────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  屏幕监控模块 (vision/screen_monitor.py) [新建]                 │
│  ├── ScreenMonitor - 定时/事件驱动截图                          │
│  ├── OCREngine - 集成选定 OCR 引擎                              │
│  └── ScreenAnalyzer - LLM 屏幕分析                              │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 推荐的 OCR 引擎选择

**首选方案：PaddleOCR**

理由：
1. 中文精度最高 (95.2%)，适合游戏场景
2. GPU 占用适中 (~1-2GB)
3. 支持文本区域定位，可用于游戏 UI 分析
4. 与现有 FunASR/Paddle 相关技术栈一致

**备选方案：RapidOCR**

理由：
1. 速度最快，实时性最佳
2. 模型最小，显存占用最低
3. ONNX Runtime，依赖简单

### 5.3 集成 LLM/TTS 的工作流

```
1. 屏幕捕获
   └─ mss 截图 → 保存临时文件

2. OCR 识别
   └─ PaddleOCR/RapidOCR → 文字 + 位置

3. 屏幕分析 (可选)
   └─ 直接送 LLM 分析文字内容

4. LLM 决策
   └─ MiniMax M2 → 结构化分析 + 建议

5. TTS 输出
   └─ GPT-SoVITS → 语音播报
```

## 八、推荐实施方案

### 8.1 方案选择

**推荐方案：PaddleOCR + MiniMax LLM + 事件驱动监控**

| 组件 | 选择 | 理由 |
|------|------|------|
| **OCR 引擎** | PaddleOCR | 中文最佳，GPU 占用适中 |
| **LLM** | MiniMax M2 (已有) | 无需额外集成 |
| **TTS** | GPT-SoVITS (已有) | 无需额外集成 |
| **监控模式** | 事件驱动 + 定时 | 节省资源 |
| **视觉分析** | 可选 (后续添加) | 当前先用纯 OCR |

### 8.2 实现优先级

```
P0 (核心功能):
├── [ ] OCR 引擎封装 (PaddleOCR)
├── [ ] 屏幕捕获模块
└── [ ] 单次截图 → OCR → LLM 分析

P1 (监控功能):
├── [ ] 监控循环 (定时/事件驱动)
├── [ ] 分析结果展示
└── [ ] 监控开关 UI

P2 (联动功能):
├── [ ] TTS 语音播报分析结果
├── [ ] 监控模式配置面板
└── [ ] 热键支持

P3 (增强功能):
├── [ ] 视觉描述 (BLIP)
├── [ ] 区域选择
└── [ ] 游戏状态识别模型
```

### 8.3 不推荐的方案

| 方案 | 不推荐原因 |
|------|-----------|
| **OmniParser** | GPU 显存需求 6-16GB，3070 Ti 8GB 不足 |
| **EasyOCR** | 中文精度不如 PaddleOCR |
| **全云端** | 实时性差，不符合本地化需求 |

## 九、总结

### 9.1 可行性评估

| 维度 | 评估 |
|------|------|
| **技术可行性** | ✅ 可行，已有类似项目实现 |
| **资源需求** | ✅ 可满足 (8GB GPU 足够) |
| **集成复杂度** | ⚠️ 中等，需新增模块 |
| **开发周期** | ⚠️ 5-8 个工作日 |

### 9.2 关键决策点

1. **OCR 引擎**：选择 PaddleOCR（中文优先）或 RapidOCR（速度优先）
2. **监控模式**：事件驱动 + 定时，平衡实时性与资源消耗
3. **视觉分析**：暂不实现，后续按需添加
4. **TTS 联动**：复用现有 GPT-SoVITS，无需额外开发

### 9.3 下一步行动

1. **立即可行**：基于 PaddleOCR 实现基础 OCR 功能
2. **快速验证**：单次截图 → OCR → LLM 分析链路
3. **迭代优化**：根据实际效果调整 OCR 引擎和提示词
